# STP v0.2.0 Document Ecosystem Audit

**Date:** 2026-04-02
**Auditor:** Claude Opus 4.6
**Scope:** All document-generating commands (`new.md`, `plan.md`, `feature.md`, `evaluate.md`) and the documents they produce (`PRD.md`, `PLAN.md`, `CONTEXT.md`, `CHANGELOG.md`, `VERSION`, `CLAUDE.md`)

---

## Methodology

Each document template was compared against industry-standard equivalents:
- **PRD.md** vs. Notion/Linear/ProductBoard PRD templates, Marty Cagan's product spec standards
- **PLAN.md** vs. Google Design Docs, ADRs, RFC format, ThoughtWorks tech radar
- **CONTEXT.md** vs. codebase documentation best practices (Backstage, Arc42)
- **CHANGELOG.md** vs. Keep a Changelog (keepachangelog.com) + Conventional Commits
- **Cross-document** vs. completeness for AI-only context reconstruction

## 1. PRD.md Audit

**Generated by:** `/stp:new` (Step 5)
**Updated by:** `/stp:feature` (Step 5, item 4 — Technical Decisions Log only)

### What It Has (GOOD)

| Element | Status | Notes |
|---------|--------|-------|
| Product description (enriched) | PRESENT | "What We're Building" section, enhanced from user input |
| Target users | PRESENT | "Who It's For" section |
| Architecture decisions with alternatives + downsides | PRESENT | Very thorough — each decision has what/why/who/vs/downside |
| Core features (user-defined) | PRESENT | Listed under "Features > Core" |
| Surfaced features (AI-identified) | PRESENT | Auth, error handling, empty states, etc. |
| Out of scope | PRESENT | Explicitly deferred items |
| Technical decisions log | PRESENT | Updated during feature building |

### What It's MISSING (GAPS)

| Gap | Priority | Industry Standard | Impact |
|-----|----------|-------------------|--------|
| **User personas** | HIGH | Notion/Linear PRDs define 2-3 personas with demographics, goals, pain points, tech savviness | Without personas, the AI makes UX decisions in a vacuum. A "freelancer tracking invoices" could be a tech-savvy developer or a 55-year-old plumber. The UI should differ dramatically. |
| **User stories with acceptance criteria** | HIGH | "As a [persona], I want [action] so that [benefit]" with testable acceptance criteria per story | Features are listed as bullets ("Ingredient CRUD") without defining WHEN the feature is "done." The Critic has no behavioral spec to evaluate against. Acceptance criteria are the bridge between product intent and test cases. |
| **Success metrics / KPIs** | HIGH | ProductBoard, Amplitude — DAU, retention, conversion rate, time-to-value, error rate targets | No way to know if the product is WORKING after launch. "Invoice app" — but is success 10 invoices sent/month? 100? What's the target time to create an invoice? Without metrics, the AI can't optimize for what matters. |
| **Non-functional requirements** | MEDIUM-HIGH | Performance targets (page load < 2s), uptime SLA (99.9%), max response time (< 500ms), concurrent user targets | PLAN.md mentions "80%+ code coverage on critical paths" but there are NO performance budgets, NO uptime targets, NO concurrent user estimates in the PRD. The plan's performance section has nothing to design against. |
| **Competitive analysis** | MEDIUM | What competitors do, how this differs, unique value proposition | Step 1 of `/stp:plan` does "Domain Research" (what existing tools include), but this lives in PLAN.md, not PRD.md. The PRD itself has zero competitive context. A product owner reading PRD.md alone can't see positioning. |
| **Priority/importance per feature (MoSCoW)** | MEDIUM | Must-have vs. Should-have vs. Could-have vs. Won't-have | Features are split into "Core" and "Included" but there's no priority within those groups. If the user runs out of budget at Milestone 2, which features can be cut? Currently impossible to decide from the PRD alone. |
| **Constraints and assumptions** | MEDIUM | Budget, timeline, team size, regulatory environment, browser support matrix | `/stp:new` asks "Just you building this, or will others join later?" but the answer doesn't appear in PRD.md. No budget constraints, no timeline, no browser/device support matrix. |
| **Risk assessment** | LOW-MEDIUM | Technical risks, market risks, dependency risks with mitigation strategies | Architecture decisions list "honest downsides" which partially covers this, but there's no consolidated risk register. |

### Severity Assessment
**PRD.md is approximately 55% complete** compared to industry-standard PRDs. The architecture decisions section is excellent (better than most PRDs), but the product specification side is thin. The AI is making UX and priority decisions without personas, success metrics, or acceptance criteria.

### What It's MISSING (GAPS)

| Gap | Priority | Industry Standard | Impact |
|-----|----------|-------------------|--------|
| **Authentication/authorization design (detailed)** | HIGH | Who can access what, role definitions, permission matrix, row-level security rules, token strategy, session management | API design says "Auth: Required" per endpoint but there's no centralized auth design. No role matrix. No permission model. No session strategy. For multi-user apps (most apps), this is the #1 source of security vulnerabilities. The Critic checks "are authenticated endpoints marked" but can't verify the auth MODEL is correct if it doesn't exist. |
| **Error handling strategy (system-wide)** | HIGH | Error taxonomy, how errors propagate (DB -> service -> API -> client), user-facing error messages, error codes, retry strategy | Milestone 3 says "Error handling across all operations" as a feature to BUILD, but the PLAN doesn't DESIGN the error strategy upfront. Without a designed error taxonomy, each feature invents its own error handling, leading to inconsistency. |
| **API error response shapes** | MEDIUM-HIGH | Standard error envelope: `{ error: { code, message, details } }`, HTTP status code mapping, pagination format | API design shows request/response for SUCCESS cases only. No error response shapes. No standard error envelope. No pagination contract. Every feature will invent its own error format. |
| **Environment strategy (dev/staging/prod)** | MEDIUM-HIGH | Environment parity, feature flags, environment-specific configs, database branching | CONTEXT.md lists env vars. Milestone 4 says "verify all env vars documented." But there's no environment STRATEGY — how many environments, how they differ, how to promote code between them, feature flag strategy. |
| **Deployment strategy** | MEDIUM-HIGH | CI/CD pipeline design, blue-green/canary strategy, rollback procedure, infrastructure as code | `/stp:new` generates `.github/workflows/ci.yml` but the PLAN doesn't design the deployment pipeline. No rollback strategy beyond database migrations. No deploy-to-production procedure. No infrastructure decisions. |
| **Monitoring/alerting strategy** | MEDIUM | What to monitor (error rate, latency, CPU, disk), alert thresholds, on-call procedure, dashboards | Milestone 3 mentions "Error tracking + monitoring setup" and "Sentry" but doesn't design WHAT to monitor or WHEN to alert. No latency budgets, no error rate thresholds, no alerting rules. |
| **Security considerations (consolidated)** | MEDIUM | OWASP Top 10 analysis, input validation strategy, CSRF/XSS prevention, CSP headers, dependency scanning | The `_standards-index.md` references excellent security standards files, but the PLAN itself doesn't include a security analysis section. Security is checked by the Critic AFTER the plan, not designed INTO the plan. The standards exist but aren't integrated into the architecture document. |
| **Scalability considerations** | MEDIUM | Expected load, horizontal vs. vertical scaling, caching strategy, CDN, database read replicas, queue systems | Zero mention. For an MVP this may be acceptable, but even a brief "designed for X concurrent users, scale path is Y" helps future decisions. |
| **Dependency list with rationale** | LOW-MEDIUM | Every npm/pip/cargo package, why it was chosen, alternatives considered, license | Architecture decisions cover major deps (framework, DB, auth) but not utility libraries. No consolidated dependency manifest with rationale. |
| **Performance requirements (response time budgets)** | LOW-MEDIUM | Per-endpoint latency targets, database query time budgets, frontend performance budgets (LCP, INP, CLS) | The standards index references `core-web-vitals.md` with targets, but the PLAN doesn't set project-specific performance budgets. "Dashboard loads in < 2s" should be in the plan, not discovered during performance optimization. |
| **Data backup/recovery strategy** | LOW-MEDIUM | Backup frequency, retention period, recovery time objective (RTO), recovery point objective (RPO) | Zero mention. If the database dies, there's no plan. |
| **State management design** | LOW | Client-side state strategy (React Context vs. Zustand vs. Redux), server state vs. client state boundary | Not addressed. For simple apps this is fine; for complex apps with real-time features, this omission causes architectural drift. |

### Severity Assessment
**PLAN.md is approximately 70% complete** compared to industry-standard technical design docs. The strongest areas are data models, API design, and feature breakdown (which are excellent and very specific). The weakest areas are cross-cutting concerns: auth design, error strategy, deployment, monitoring, and security are either missing or deferred to "build time" when they should be designed upfront.

## 5. Cross-Document Analysis

### 5A. Can the AI Reconstruct Full Project Context from Documents Alone?

**MOSTLY YES, with critical gaps:**

| Document | Provides | Missing for Full Reconstruction |
|----------|----------|--------------------------------|
| PRD.md | What to build, for whom, architecture decisions | User personas, acceptance criteria, success metrics |
| PLAN.md | How to build it, data models, API, milestones | Auth design, error strategy, deployment procedure |
| CONTEXT.md | What exists now, file map, schema, endpoints | Build/run commands, dependencies, known issues |
| CHANGELOG.md | What was built, when, decisions, quality grades | Nothing critical — good history document |
| CLAUDE.md | How to build (patterns, standards, conventions) | N/A — generated from templates, comprehensive |
| VERSION | Current version number | N/A — simple and complete |

**Critical reconstruction gaps:**
1. A fresh session cannot RUN THE APP without discovering commands (CONTEXT.md gap)
2. A fresh session doesn't know the auth model (PLAN.md gap)
3. A fresh session doesn't know what "done" looks like per feature (PRD.md gap — no acceptance criteria)
4. A fresh session doesn't know what's currently broken (CONTEXT.md gap — no known issues)

### 5B. Information That Exists Only in Conversation

| Information | Where It Lives | Should Also Be In |
|-------------|----------------|-------------------|
| User's detailed answers to product questions | Conversation only (lost on /clear) | PRD.md — the enriched description captures SOME but not all nuance |
| Why specific alternatives were rejected | Conversation + PRD.md architecture section | PRD.md has this — GOOD |
| Edge cases discovered during building | Conversation + possibly commit messages | PLAN.md (edge cases section exists) + CHANGELOG.md |
| Failed approaches during debugging | Conversation only (unless /stp:pause is used) | `.stp/handoff.md` captures this IF pause is run — but volatile |
| User's verbal approval of plan | Conversation only | No record of user sign-off exists in any document |

**Key gap:** The `/stp:pause` command creates `.stp/handoff.md` with "Problems / Things That Didn't Work" — but this file is DELETED after the feature completes (feature.md Step 5, item 6). Failed approaches are permanently lost. They should be preserved somewhere (CHANGELOG.md "Decisions Made" partially covers this, but not systematically).

### 5C. Circular Dependencies

**No true circular dependencies found.** The reading order is well-defined:
1. PRD.md (what) -> PLAN.md (how) -> CONTEXT.md (current state) -> CHANGELOG.md (history)

However, there is a **soft circularity** between PLAN.md and PRD.md:
- PLAN.md Phase 1 (Domain Research) discovers features that should be in the PRD
- But PLAN.md says "present to user" not "update PRD.md"
- Result: PLAN.md's Domain Research section may contain product requirements that PRD.md doesn't have
- This means PRD.md is NOT the single source of truth for product requirements

### 5D. Reading Order

**The reading order IS clear and documented:**
- Critic reads: VERSION -> CONTEXT.md -> CHANGELOG.md -> PRD.md -> PLAN.md -> CLAUDE.md
- Feature reads: PLAN.md -> CLAUDE.md -> existing code
- New session reads: `.stp/handoff.md` (if exists) -> CONTEXT.md -> PLAN.md

**Issue:** There's no "START HERE" document. A fresh AI session (or human developer) has to know to read CONTEXT.md first. A simple `README-AI.md` or a header in CONTEXT.md saying "Read this first" would help.

## 8. Positive Findings (What Pilot Does BETTER Than Most)

Credit where due — several aspects of Pilot's document ecosystem are genuinely better than industry standard:

1. **Architecture decisions with honest downsides** — Most PRDs list decisions without alternatives. Pilot's format (what/why/who-uses/vs-alternatives/downside) is exceptional.

2. **Living documents by design** — CONTEXT.md full refresh at milestones, CHANGELOG per feature, PLAN.md updated on completion. Most projects let docs rot.

3. **Critic as separate evaluator** — Using a different AI model to evaluate prevents self-congratulation bias. The 6-criteria framework with file:line evidence is rigorous.

4. **Feature-level version tracking** — Semantic versioning per feature (not just per release) creates a granular history. The VERSION + CHANGELOG + git tag trio is well-designed.

5. **Standards-as-code via .stp/references/** — Security, accessibility, performance, and production standards are FILES the AI reads before implementing, not just guidelines in a document.

6. **TDD enforcement in the workflow** — Tests before code, enforced at the process level, with the Stop hook blocking completion until tests pass. This is more rigorous than most professional teams.

7. **Handoff document for session continuity** — `.stp/handoff.md` with failed approaches and verification commands is excellent session management.

8. **Milestone-boundary full context refresh** — CONTEXT.md complete rewrite at milestones catches drift that incremental updates miss. This is a clever design.
