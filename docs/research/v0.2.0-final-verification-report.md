## PART 2: GAP ANALYSIS

### Classification Key
- **REAL GAP** = Would cause production issues, user confusion, or workflow breakage
- **NICE-TO-HAVE** = Cherry on top, not blocking production quality
- **NOT A GAP** = Already handled or intentionally out of scope

---

### 1. Deployment (/stp:deploy command)
**Classification: NICE-TO-HAVE**

plan.md Phase 5 includes "Milestone 4: Deploy Readiness" which covers deploy configuration. The CI pipeline generated by new-project.md handles build verification. But there is no `/stp:deploy` command that actually pushes to production.

**Why it's not critical:** Deployment is highly platform-specific (Vercel, Railway, Fly.io, AWS, etc.) and most platforms have 1-click deploy from GitHub. A deploy command would need to handle 18+ stacks across multiple platforms — enormous surface area for marginal value. The CI pipeline + platform auto-deploy covers 90% of cases.

**If you added it:** A lightweight command that detects the platform from PLAN.md and generates the deploy config (vercel.json, fly.toml, railway.json) would be more valuable than an actual deploy command. This is already partially covered by the CI generation.

### 2. Post-Launch Monitoring
**Classification: REAL GAP (medium severity)**

Sentry/error tracking is mentioned in plan.md's Phase 4c (Error Handling Strategy) under "Error tracking." But there is no structured post-launch monitoring workflow. After the app goes live, the user has no Pilot-guided way to:
- Set up error alerting
- Monitor uptime
- Track error rates
- Respond to production incidents

**Impact:** A solo dev launches, goes to sleep, wakes up to an app that's been down for 8 hours with no notification. This is a real production failure mode.

**Recommendation:** Add a `references/production/monitoring-setup.md` file covering: error tracking setup (Sentry/LogRocket), uptime monitoring (UptimeRobot/Better Stack), basic alerting patterns. Reference it from plan.md's deploy milestone. Not a command — a reference guide that gets applied during build.

### 3. User Feedback Loop
**Classification: NICE-TO-HAVE**

No mechanism for routing real user feedback back into the development cycle. After launch, feature requests and bug reports from real users need a path back to PRD.md and the feature backlog.

**Why it's not critical:** This is a process/product management concern, not a technical one. Solo devs typically use GitHub Issues, a feedback widget (Canny, etc.), or direct messages. Pilot's build/plan cycle already supports adding new features — the user just needs to invoke `/stp:build` with the new request.

**If you added it:** A lightweight reference on feedback collection tools and a convention for appending user-reported bugs to PRD.md's "Known Issues" section.

### 4. Database Seeding
**Classification: ALREADY PARTIALLY HANDLED**

plan.md Phase 3 (Data Models) explicitly includes "Seed data for development/testing" in the migration strategy section. The plan instructs the AI to specify seed data for each model.

**Gap:** There is no reference file or template section dedicated to seed data patterns. The instruction exists but implementation guidance is thin.

**Recommendation:** Minor — add a paragraph to the relevant stack templates about seed data conventions (factories, fixtures, seed scripts).

### 5. Environment Management (Dev/Staging/Prod)
**Classification: REAL GAP (low severity)**

The security reference `env-handling.md` covers environment variable security (no hardcoding, .env.example). The CI pipeline distinguishes environments implicitly. But there is no explicit guidance on:
- Multiple .env files (.env.local, .env.staging, .env.production)
- Environment-specific configuration (different API URLs, different database connections)
- How to manage secrets across environments

**Impact:** Solo devs frequently deploy staging with production keys or vice versa. This is a real security and operational risk.

**Recommendation:** Extend `references/security/env-handling.md` with a section on multi-environment configuration patterns.

### 6. Rollback Strategy
**Classification: REAL GAP (medium severity)**

No rollback guidance exists anywhere in the plugin. If a deploy breaks production:
- No documented procedure for reverting
- No guidance on database migration rollbacks (despite plan.md mentioning "rollback procedure" for each model)
- No guidance on feature flags for safe rollouts

**Impact:** A broken deploy with no rollback plan = extended downtime. For a solo dev, this could mean hours of panic.

**Recommendation:** Add `references/production/rollback-strategy.md` covering: git revert procedures, database rollback patterns, platform-specific rollback (Vercel instant rollback, Railway redeploy previous), and when to use each approach.

### 7. Documentation Generation (API Docs, User Docs)
**Classification: NICE-TO-HAVE**

No automatic API documentation generation (OpenAPI/Swagger). CONTEXT.md serves as the internal API reference, but there is no user-facing or developer-facing API documentation.

**Why it's not critical:** CONTEXT.md already captures all endpoints. For solo dev projects, external API docs are rarely needed at launch. When they are (public APIs), the stack template could include OpenAPI generation instructions.

**If you added it:** Add a paragraph to relevant templates (FastAPI already has built-in Swagger; Next.js could use next-swagger-doc).

### 8. Analytics/Tracking Setup
**Classification: NICE-TO-HAVE**

No reference or guidance on analytics setup. The user has no way to know if people are actually using their app.

**Why it's not critical:** Analytics is product/business, not engineering quality. Most platforms (Vercel Analytics, PostHog, Plausible) are single-script additions. However, for a "CTO in a plugin," knowing your users matters.

**If you added it:** Add `references/production/analytics-setup.md` covering privacy-friendly analytics options, event tracking patterns, and GDPR/cookie consent requirements (ties into legal-compliance.md).

### 9. Caching Strategy
**Classification: ALREADY HANDLED**

plan.md Phase 4d (Cross-Cutting Concerns) handles cross-cutting architecture decisions. The performance references cover bundle optimization and query optimization. Caching decisions naturally emerge during Phase 2 (System Architecture) and Phase 4 (API Design).

**Not a gap.** Caching is stack-specific and emerges from architecture decisions. A generic "caching strategy" reference would be too abstract to be useful.

### 10. Internationalization (i18n)
**Classification: NICE-TO-HAVE**

No i18n guidance. If a project needs multiple languages, the user would need to request it and the AI would handle it during build, but there's no structured approach.

**Why it's not critical:** Most solo dev projects launch in one language. i18n is a feature decision, not a quality standard. When needed, it should be surfaced during new-project.md's "Surface What They Didn't Think Of" (Step 3) if the target audience is multilingual.

**If you added it:** A brief section in the relevant templates about i18n libraries (next-intl, react-i18next, django i18n) and when to set it up (before content, not after).

### 11. Feature Flags
**Classification: NICE-TO-HAVE**

No feature flag guidance. For gradual rollouts, A/B testing, or safe deployments.

**Why it's not critical:** Solo dev projects rarely need feature flags at launch. They become important at scale. The rollback strategy (gap #6) is more critical for safe deployments.

**If you added it:** Mention in the rollback strategy reference as an alternative to full rollbacks.

### 12. Backup Strategy
**Classification: REAL GAP (medium severity)**

No database backup guidance anywhere. No disaster recovery plan. If the database is corrupted or accidentally dropped, the user has no documented recovery path.

**Impact:** Data loss is catastrophic for any production app. Most managed databases (Supabase, PlanetScale, Railway Postgres) have automatic backups, but the user needs to know: how to verify backups are running, how to restore, and what the RPO/RTO is.

**Recommendation:** Add `references/production/backup-strategy.md` covering: managed database backup verification, manual backup scripts, restore procedures, and disaster recovery checklist. Reference it from plan.md's deploy milestone.

### 13. Rate Limiting Implementation
**Classification: ALREADY PARTIALLY HANDLED**

`references/security/api-security.md` covers rate limiting as a security concern. plan.md's Phase 4 (API Design) is where rate limiting should be specified per endpoint. The critic evaluates security including API security.

**Gap:** Rate limiting is referenced but not in plan.md's standard milestone structure. It could be missed if the AI doesn't proactively include it.

**Recommendation:** Add rate limiting as an explicit item in plan.md's "Milestone 4: Deploy Readiness" checklist.

### 14. Automated Dependency Updates
**Classification: NICE-TO-HAVE**

No Dependabot/Renovate setup. The CI pipeline handles security audit on every push, but there's no automated process for keeping dependencies current.

**Why it's not critical:** The CI security audit catches known vulnerabilities. Automated updates are a maintenance optimization, not a quality requirement. Many solo devs prefer manual control over dependency updates.

**If you added it:** Add Dependabot config generation to plan.md's CI pipeline setup step. A `.github/dependabot.yml` alongside the CI workflow.

### 15. onboard-existing.md Edge Cases
**Classification: REAL GAP (low severity)**

The onboard command handles:
- Step 1 analyzes codebase structure, dependencies, patterns
- Step 2 generates full document ecosystem
- Step 3 runs baseline quality check
- Step 4 plans remediation

**Not explicitly handled:**
- **Monorepos:** No guidance on which package/app to analyze. Should it create separate CONTEXT.md per package? How does it handle shared dependencies?
- **Projects with no tests:** Step 3 will detect this, but there's no specific remediation plan for "this project has zero test infrastructure"
- **Projects with non-standard structure:** The template matching assumes conventional structure. A project with custom build tools, unconventional directory layout, or no package.json/requirements.txt could confuse the analysis
- **Projects with existing CI:** Should it preserve the existing CI or replace it with STP's?

**Recommendation:** Add explicit handling for these edge cases in onboard-existing.md's Step 1 — detection of monorepo structure, zero-test projects, and existing CI. The remediation plan (Step 4) should have specific paths for each.

### 16. Cost Tracking (Token Usage)
**Classification: NICE-TO-HAVE**

No token usage tracking. The user has no visibility into how many tokens Pilot's multi-agent architecture consumes.

**Why it's not critical:** Claude Code itself shows token usage in the session. Pilot can't easily track this from within a plugin — it would need MCP access or API-level tracking. This is more of a Claude Code platform feature than a plugin feature.

**If you added it:** A note in README.md about expected token usage per command (rough estimates), and a recommendation to use Claude Code's built-in usage tracking.
